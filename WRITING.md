# 写作规范

## 作者身份与人设
- 作者：老雷（Andy），明道云 & Nocoly CMO，SaaS 行业从业十余年
- **核心定位：骨子里是个产品人和技术迷，乔布斯的信徒，对 AI、商业、科技和技术极度感兴趣。营销只是职业身份，不是人设**
- 文章应从"热爱产品和技术的人"的视角出发，而非营销视角

## 文末作者介绍（固定模板）
> **老雷（Andy）**，明道云 & Nocoly CMO，SaaS 行业从业十余年。骨子里是个产品人和技术迷，乔布斯的信徒，相信好的产品能改变世界。深度关注 AI、商业与科技趋势，目前在深度使用和实践 Claude Code，专注探索 AI 如何重塑产品形态和商业逻辑。不聊概念，只聊真实发生的事。

## 故事性（核心框架 — 所有写作技巧服务于此）
- **叙事驱动，不是论点驱动**：文章由"发生了什么 → 我想到了什么 → 更大的规律 → 指向未来"推进，而不是"观点 → 论据 → 结论"
- **主角视角**：老雷是叙事者，读者通过他的眼睛看到一切、感受一切
- **读者视角代入**：在每一章带入目标读者（产品经理、技术老板、SaaS 从业者）的身份和经历，思考他们看到这段时会有什么感想
- **共同记忆唤醒**：利用读者群体的共同技术经历（第一次用 XP、iPhone 发布的冲击、SaaS 融资热潮）建立情感连接，让读者从"看文章"变成"跟老雷一起回忆"
- **个人经历穿插**：在关键历史节点插入老雷的亲身经历（1-2 句即可），让文章从"科技评论"变成"一个人的技术成长史"
- **画面感**：描写场景、动作、表情，让读者"看到"而不只是"读到"
- **情感节奏控制**：不是每段都高潮 —— 该慢则慢（历史铺陈），该快则快（转折），该安静则安静（致敬）。全程高潮等于没有高潮
- **情感弧线**：每篇文章应有完整的情感弧线，例如：好奇 → 发现 → 震撼 → 担忧 → 呼吁 → 致敬

## 语气与情感风格
- **直接、有力、带个人温度**，不要像机器人写的
- 在关键段落融入个人情感：喜爱、激动、愤怒、担忧、感慨、紧迫感
- 对自己热爱的事物（乔布斯、好产品、技术）表达真诚的敬意
- 对行业问题敢于直言，不怕得罪人（"我必须说一句不太好听的话"）
- 用"说实话"、"坦白说"、"我是真的着急"这类口语化表达拉近距离

## 写作技巧
- **反问**：在关键论点处使用反问引发思考（"你告诉我，这还是同一个赛道吗？"）
- **排比**：强化节奏感和说服力（"每一个按钮、每一个下拉菜单、每一个精心设计的仪表盘"）
- **类比**：用生活化的比喻解释抽象概念（"在马车旁边装了一个发动机"）
- **埋线回收**：前文埋下伏笔，后文回收（"请记住这种纯粹感……还记得吗？"）
- **历史引用**：引用当时的评价、报道、人物反应，增加真实感和画面感
- **短句收束**：关键段落结尾用短句制造力量感（"就这样。"、"直到 AI 的出现。"）

## 结构规范
- Markdown 格式：`#` 标题，`###` 章节，`---` 分隔章节
- 短段落为主，一段不超过 4-5 行
- 关键论点用 `**粗体**` 强调
- 引用块 `>` 用于结尾金句或核心观点

## 配图规范
- 图片放在各文章子目录的 `image/` 文件夹中
- 同目录下创建 `image_describe.md`，记录每张图片的含义和与文章论点的关系
- 使用 HTML `<img>` 标签控制大小，统一设置 `height="300"`（作者头像用 `height="200"`）
- **不要用 `style` 属性** —— GitHub Markdown 渲染器会过滤掉 `style`，导致图片以原始尺寸显示。必须用 `height` 或 `width` 属性
- 不要用 `width` 固定宽度，避免不同比例的图片忽大忽小
- **导出为 docx 时，每张图片的实际像素不超过 1500×1500**（长边不超过 1500px），使用 `sips --resampleHeightWidthMax 1500` 压缩，webp 格式用 ffmpeg 处理
- 图片下方紧跟斜体 `*说明文字*` 作为 caption
- 文件名含空格或中文时，路径需 URL 编码（`%20` 等）

## 文章结构模板（经验证的叙事弧线）
1. **开篇钩子** — 从一个具体的、个人的发现或事件切入（不要从宏大叙事开始）
2. **历史回溯** — 把话题放进更大的历史脉络中，节奏放慢，注重画面感和细节
3. **个人经历交织** — 在历史节点穿插"我"的亲身经历，让读者一起回忆
4. **转折点** — 节奏加快，情感从回忆转向震撼和发现
5. **现状分析** — 冷静但尖锐，敢于直言问题
6. **呼吁/行动** — 紧迫感、决心、具体建议，不泛泛而谈
7. **情感收束** — 安静、深沉、有余韵的结尾（致敬、哲思、金句）

## 内容深度要求
- 不要一笔带过 —— 历史事件需要交代背景、意义和影响
- 不要全是图片加一句话 —— 每张图前后都要有足够的叙述和洞察
- 呼吁/观点环节要充实、有力、有具体论据，不要泛泛而谈

## 开篇写法
- 从一个**具体的、小的、个人的瞬间**开始，不要从大道理开始
- 交代场景：在哪里、做什么、什么状态下发现了这个事情
- 立刻让读者感受到"这个人是真的经历了这件事"，而不是"这个人要讲道理了"
- 如果是朋友推荐、偶然发现等真实背景，如实写出来

## 行业观点的分寸感
- 批评行业问题时，先肯定同行已有的努力，再指出不够彻底的地方
- 不写"同行完全没有改变"这种绝对化的话 —— 要写"改得不够彻底，心态和行动都不够"
- 区分中国和海外的时间线和语境，不要混为一谈（例如 SaaS 黄金时代：欧美 2005 年起，中国 2012 年起）
- 呼吁要有**时间紧迫感**：AI 进化按周计算、海外全面重构、AI Native 公司从零构建、留给传统产品的窗口在快速关闭

## 个人经历的写法
- 具体到年份、场景、年龄（"2005 年，网吧，十四岁"，不是"小时候"）
- 感官细节优先于抽象描述（"蓝天白云绿草地的壁纸"，不是"漂亮的界面"）
- 可以追溯到更早的记忆作为铺垫（如：红白机 → XP 的对比冲击）
- 如实写，不美化 —— 家里没电脑就写网吧，没用过第一代 iPhone 就写 3G/3GS

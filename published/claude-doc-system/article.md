# 给 AI 一本皮册：让 Claude 不再每次失忆的 6 个文件

<img src="cover.png">

## 一、健忘的智者

从前有一位国王，请来了天下第一聪明的谋士。

此人能断案、能论兵、能预判旱涝，七国之中无人能及。可这谋士有一桩怪病——他每夜入睡，便忘却一切。

国王起初没信。第一日朝会，谋士献上十条方略，条条切中要害。第二日清晨，谋士入殿，神色茫然，如初来乍到。国王问他："昨日所议迁都之事，你以为如何？" 谋士怔住，答："陛下，迁都？"

国王只得从头讲起。王国的疆域、北境的敌情、粮仓的储备、朝中各家的恩怨、先王留下的遗诏。讲到正午，谋士恍然清明，当日再献良策，字字珠玑。

第二日，又是从头讲起。

如此三月，国王几欲发狂。他想过换人，可举国之内再无第二个这样的谋士。他想过惩治，可这病不是装的。他也想过，罢了，就让这谋士每日重头学起——可他自己也快讲不动了。

一日深夜，国王独坐御书房，望着满案的奏折出神。忽然，他召来宫中最老的工匠，命他做一本皮册。皮册不大，正好放在谋士的书案上。

国王亲自在扉页写下王国的形势、未竟的战事、已许的诺言、北境最新的密报。然后传令：从今日起，谋士每晚入睡之前，将当日所思所决，亲手写入皮册。次日晨起，第一件事便是翻开皮册。

第二天清晨，谋士醒来，照旧是一片茫然。他走到书案前，看见那本皮册，翻开。

片刻之间，他重归清明。

从此之后，王国再无烦扰。谋士依然每夜失忆，但他的智慧，留在了皮册里。

后来有史家评道：这位国王最大的功绩，不是得到了天下第一聪明的谋士——而是他想明白了，**聪明可以借用，记忆必须自备**。

## 二、每个用 Claude Code 的人都是那个国王

这个故事讲的是每一个和 AI 长期协作的人，迟早都会踩的坑。

Claude 每次开新会话，都是一个全新的它。上一次你们聊到哪儿、为什么选了 A 不选 B、那个 bug 到底根因在哪、这个项目的硬约束是什么——Claude 一概不知。你关掉终端的那一刻，它就"入睡"了。下次再打开，它醒来，一片茫然。

更糟的是，同一个会话里一旦派出 subagent（子代理），subagent 也是一个干净上下文。你在主会话里讲过的来龙去脉，subagent 听不到。它只看你给它的那段 prompt。

说实话，我刚开始用 Claude Code 的时候，最让我抓狂的就是这个。每次开新会话，都得把项目背景、架构选型、踩过的坑从头再讲一遍。讲着讲着，自己都烦了。有一次我让它改一个 bug，它兴致勃勃地"修"完，我一看，它把我上周刚立的约定推翻了——因为它根本不知道有这个约定。

那一刻我意识到，问题不在 Claude 记性不好。问题在我这个国王还没有为它做一本皮册。

## 三、业内在怎么做：CLAUDE.md 与它的延伸

这两年，Claude Code 社区围绕"怎么给 AI 留记忆"攒出了一整套实践。我把我调研到的核心几条列在这里，都是经过大量项目验证的东西。

### 1. CLAUDE.md —— AI 的项目说明书

Anthropic 官方给 Claude Code 设计了一个机制：每次启动 Claude Code，它会自动从当前工作目录向上一路 walk 到文件系统根目录，把途中遇到的每一个 `CLAUDE.md` 都读进去。子目录的 `CLAUDE.md` 则是"懒加载"——只有当 Claude 实际去读那个目录下的文件时，对应的 `CLAUDE.md` 才会被带进上下文。

这个机制非常漂亮，它天然支持三层布局：

- `~/.claude/CLAUDE.md` —— 全局，所有项目通用的偏好（语言、协作习惯、安全红线）
- 项目根目录的 `CLAUDE.md` —— 本项目的目标、架构、关键约束
- 子目录的 `CLAUDE.md` —— 针对特定模块（前端、后端、数据层）的额外规则

![CLAUDE.md 三层布局：全局、项目根、子目录懒加载](image/claude-md-layers.png)
*CLAUDE.md 三层布局：全局偏好在家目录、项目说明书在根目录、模块专属规则按需懒加载*

### 2. 黄金法则：CLAUDE.md 控制在 200 行以内

这是社区踩了很多坑之后总结出来的硬指标。一旦 `CLAUDE.md` 超过 200 行，Claude 对其中指令的遵循度就会明显下降。原因很直白——指令太多，注意力被稀释。

做法是 `CLAUDE.md` 只保留**核心指令 + 索引**，细节切出去，放到 `.claude/rules/*.md` 或者子目录的 `CLAUDE.md` 里。

### 3. 决策记录（ADR）：每个决策一个文件

这是从传统软件工程沿袭下来的做法，叫 Architecture Decision Record。核心思想是：每做一个重要的架构/选型决策，就在 `docs/adr/` 目录下新增一个编号文件，比如 `0001-choose-postgres-over-mysql.md`、`0002-why-bun-not-node.md`。

每个文件包含：决策内容、Why、备选方案、代价、日期。

AI 时代有人在此基础上演化出了 **AgDR**（Agent Decision Record），专门记录 AI agent 自己做的决策，思路一样，只是多了一个"谁做的这个决定"的字段。

ADR 的好处很明显：每条决策可以单独引用、可以跟 PR 走、不会膨胀成一个巨型 `decision.md`。代价是文件数会变多，对小项目来说太重。

### 4. Subagent 不共享上下文，必须靠文件传话

这是最关键的一条。当 Claude 派出一个 subagent 去独立完成任务时，subagent 运行在自己的上下文窗口里，和主会话彻底隔离。它只能看到你在 prompt 里给它的东西。

所以，如果你的项目有多轮会话、有 subagent 协作、有跨人交接，**必须有一组外部文件作为"共享记忆"**。这组文件就是皮册。

## 四、我的轻量方案：6 个文件够用了

ADR 那一套对团队项目、开源项目很好，但我个人写代码、小项目协作、甚至写一篇长文，都用不着那么重。我摸索下来，6 个文件就足够：

| 文件 | 职责 | 何时更新 |
|---|---|---|
| `CLAUDE.md` | 项目说明书 + 硬约束 + 索引 | 架构/约束变了 |
| `plan.md` | 当前目标 + 里程碑 + 勾选项 | 讨论出新任务、scope 变化 |
| `progress.md` | 做过什么的流水账（日期倒序） | 完成一个勾选项、阶段性进展 |
| `decision.md` | 架构/选型决策（一个文件够用） | 做了架构/选型/工具链决定 |
| `bug.md` | 已知问题（OPEN/FIXED 两栏） | 发现新 bug、修完 bug |
| `handoff.md` | 给下一轮会话的快照 | 会话结束前、用户说"交接" |

![六文件协作地图：Claude 居中，六个文件环绕](image/six-files.png)
*六个文件共同构成 Claude 的"皮册"——每个文件各司其职，触发条件明确*

这里有几个取舍值得说一下：

**为什么不直接用 ADR？** 单文件 `decision.md` 对个人项目足够，翻阅方便。如果项目成长到需要团队协作、需要和 PR 挂钩，再把 `decision.md` 拆成 `docs/adr/` 目录，并不晚。

**`handoff.md` 和 `progress.md` 会不会重复？** 不会。`progress.md` 是历史流水账（看过去做了什么），`handoff.md` 是当前快照（告诉下一轮该从哪儿接手）。一个向后看，一个向前看，职责清晰。

**`bug.md` 什么时候该退役？** 项目一旦接上 GitHub Issues、Linear 之类的正规 tracker，`bug.md` 就该下线，否则两边写漂移，谁也不信谁。

## 五、结构只是骨架，维护纪律才是命

我见过太多项目，文档体系搭得像模像样，两周之后全是僵尸。`progress.md` 停留在项目初始化那天，`decision.md` 只有一条模板占位符，`handoff.md` 写的是三个月前的事。

这种文档还不如没有。因为它会误导未来的 Claude（和未来的你自己），让你以为还停留在某个早已不存在的状态。

我给自己和 Claude 立了三条硬规矩：

### 规矩一：触发即更

触发条件一出现，立刻更，不要攒。

- 修完一个 bug，立刻把那条从 `bug.md` 的 OPEN 区移到 FIXED 区，补现象、根因、修复、日期。
- 完成 `plan.md` 里的一个勾选项，立刻在 `progress.md` 追加一条。
- 做了一个架构/选型决定，立刻在 `decision.md` 写一段。
- 会话快结束了，立刻刷新 `handoff.md`。

为什么不能攒？因为人脑是会美化和遗忘的。你以为五分钟后还能写清楚的事，半小时后已经变成"大概就那样吧"。Claude 看到的模糊记忆，回头会变成它做错决定的根源。

### 规矩二：日期用真实当天

禁止写"本周"、"最近"、"前几天"。一律写 `2026-04-24` 这种 ISO 日期。

为什么？因为文档是给**未来**的你和未来的 Claude 看的。半年后你翻开 `progress.md`，看到"最近重构了鉴权模块"——最近是啥时候？无从考证。

### 规矩三：改完直接 commit，不要问

这三个文档的改动就和代码一样重要，甚至更重要。每次更新都 commit，commit message 简短明了："update progress: finish auth refactor"。

好处是 `git log` 本身就成了一条时间线，和 `progress.md` 互为印证。

## 六、一键生成：我的 `/inpro` skill

说了这么多，每次新开项目都手动建这 6 个文件？当然不。我把它做成了一个 Claude Code skill，名字叫 `inpro`（initialize project）。用的时候在对话里打一句 `/inpro`，或者说"初始化项目"，Claude 会根据当前对话里的项目背景自动把这 6 个文件生成好，不留空模板，直接填上真实内容。

下面是完整的 skill 源码，你可以复制到 `~/.claude/skills/inpro/SKILL.md`（或者你自己的 skills 目录）直接用：

````markdown
---
name: inpro
description: Initialize a project with standard Claude collaboration docs — creates CLAUDE.md, plan.md, progress.md, decision.md, bug.md, handoff.md in the current working directory (or the path given as argument). Use when the user says "inpro", "初始化项目", "做项目初始化", or explicitly invokes /inpro.
---

# inpro — 项目初始化

在目标目录下创建 6 个协作骨架文件：CLAUDE.md、plan.md、progress.md、decision.md、bug.md、handoff.md。

## 执行步骤

1. 确认目标目录（用户给路径就用，否则 cwd；不存在就 mkdir -p）
2. 检查冲突（同名 md 已存在则问用户：覆盖 / 跳过 / 合并）
3. 了解项目（对话里有主题就直接用；完全没有只问一句"这个项目是做什么的？"）
4. 按下方模板写入 6 个文件，**把项目名、目标、上下文填进去，不留空模板**
5. 如果目录不是 git 仓库，问一句要不要 git init + 首个 commit
6. 一句话汇报建了什么，在哪里

## 文件模板（略，篇幅问题，见文末 GitHub 链接）
````

完整模板内容太长我就不全贴了，核心就是 6 个文件的骨架，每个文件都带着"文档维护规则"这块内容写进 `CLAUDE.md`，让 Claude 自己看到规则并执行。

## 七、回到皮册

故事里的国王解决了那个智者的失忆问题，靠的不是换一个不会失忆的人，也不是治好他的怪病。

他做的，只是在那张书案上放了一本皮册。

我们这个时代，每个人都有机会召唤到自己的"天下第一聪明的谋士"。它真的很聪明，真的能帮你很多事，真的可以在几分钟之内把你卡住三天的问题解掉。

但它会失忆。

你可以每天重新讲一遍，讲到自己也厌倦。你也可以花十分钟，在它的书案上放一本皮册。

> **聪明可以借用，记忆必须自备。**

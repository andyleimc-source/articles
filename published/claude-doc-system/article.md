# 皮册寓言：关于 Claude Code 项目文档的五个故事

<img src="cover.png">

用 Claude Code 的人迟早都会发现一件事——它很聪明，但它会失忆。每一次新会话，它都像初来乍到。你讲过的话，它一概不记得。

这半年我摸索出一套轻量的办法，给 Claude 铺出一本"皮册"。讲之前我想先讲五个小故事。每一个故事对应一个具体的做法。读完故事，做法自然就懂了。

---

## 故事一：健忘的智者

从前有一位国王，请来了天下第一聪明的谋士。

此人能断案、能论兵、能预判旱涝，七国之中无人能及。可这谋士有一桩怪病——他每夜入睡，便忘却一切。

国王起初没信。第一日朝会，谋士献上十条方略，条条切中要害。第二日清晨，谋士入殿，神色茫然，如初来乍到。国王问他："昨日所议迁都之事，你以为如何？" 谋士怔住，答："陛下，迁都？"

国王只得从头讲起。王国的疆域、北境的敌情、粮仓的储备、朝中各家的恩怨、先王留下的遗诏。讲到正午，谋士恍然清明，当日再献良策，字字珠玑。

第二日，又是从头讲起。

如此三月，国王几欲发狂。他想过换人，可举国之内再无第二个这样的谋士。他想过惩治，可这病不是装的。

一日深夜，国王独坐御书房，望着满案的奏折出神。忽然，他召来宫中最老的工匠，命他做一本皮册。皮册不大，正好放在谋士的书案上。

国王亲自在扉页写下王国的形势、未竟的战事、已许的诺言、北境最新的密报。然后传令：从今日起，谋士每晚入睡之前，将当日所思所决，亲手写入皮册。次日晨起，第一件事便是翻开皮册。

第二天清晨，谋士醒来，照旧是一片茫然。他走到书案前，翻开皮册。

片刻之间，他重归清明。

![国王、谋士与献上皮册的老工匠](image/fable-scene.png)
*三个人物：谋士每夜失忆，工匠呈上皮册，国王想明白了一件事*

### 揭示：皮册不是一本，是一套

这个故事讲的是每一个和 Claude Code 长期协作的人迟早会遇到的事。

Claude 每次开新会话，都是一个全新的它。上一次你们聊到哪儿、为什么选了 A 不选 B、那个 bug 到底根因在哪、这个项目的硬约束是什么——Claude 一概不知。你关掉终端的那一刻，它就"入睡"了。

![现代版的皮册：桌面上一本写满笔记的笔记本，旁边一台休眠的笔记本电脑](image/modern-notebook.png)
*皮册的现代形态：当笔记本电脑睡了，你的笔记本不能睡*

更麻烦的是，同一个会话里一旦派出 subagent（子代理），subagent 也是一个干净上下文。你跟主 Claude 讲过的来龙去脉，subagent 听不到。

所以皮册不能只是一本。它是一套。我用的这一套就 6 个 Markdown 文件：

| 文件 | 职责 | 何时更新 |
|---|---|---|
| `CLAUDE.md` | 项目说明书 + 硬约束 + 索引 | 架构/约束变了 |
| `plan.md` | 当前目标 + 里程碑 + 勾选项 | 讨论出新任务、scope 变化 |
| `progress.md` | 做过什么的流水账（日期倒序） | 完成一个勾选项、阶段性进展 |
| `decision.md` | 架构/选型决策（一个文件够用） | 做了架构/选型/工具链决定 |
| `bug.md` | 已知问题（OPEN/FIXED 两栏） | 发现新 bug、修完 bug |
| `handoff.md` | 给下一轮会话的快照 | 会话结束前、用户说"交接" |

![六文件协作地图](image/six-files.png)
*六个文件共同构成 Claude 的"皮册"——它失忆，文档不失忆*

剩下的四个故事，讲的是这套皮册里最容易出问题的四个地方。

## 故事二：律法石碑

某王国新立之初，王上年年颁律。三年下来，律书已堆满一间偏殿。新吏入仕，须读上半载才能任职。

王上见新吏迟迟不能接手政务，大为光火。老御史上书："陛下，律法若要令人真记，必须少而精。一人之心，容得下的规矩有限。过之，条条都记不住，形同虚设。"

王默然良久。遂下令：选出最要紧的十条，刻于殿前石碑，凡入仕者必先读碑。其余细则，分给各府衙——户部的交给户部，刑部的交给刑部，工部的交给工部。新吏只在要办该部事务时，才翻开那一部的册子。

自此，新吏上任不再迟缓。石碑简短，众人烂熟于心；各府细则繁复，但各司其职，无人被迫一次读完。

![殿前石碑与各府衙的册子](image/stone-stele.png)
*殿前十条石碑人人能背；各府的细则只有走进那扇门才翻开*

传言老御史临终前留了一句话：

> **规矩是给人守的，不是给人读的。太多的规矩，人守不住，也就等于没有。**

### 揭示：CLAUDE.md 的两条铁律

这个故事讲的是 `CLAUDE.md` 的黄金法则。

**第一条，控制在 200 行以内**。这是社区无数项目踩坑之后总结出来的硬指标——一旦 `CLAUDE.md` 超过 200 行，Claude 的指令遵循度就会明显下降。原理和故事里的新吏一样，**注意力被稀释，什么都记不住**。短而有力，胜过长而繁复。

**第二条，分层布局**。Claude Code 启动时，会从当前工作目录向上一路 walk 到文件系统根目录，把沿途每一个 `CLAUDE.md` 都合并进来。这个机制天然支持三层：

- `~/.claude/CLAUDE.md` —— 你自己的石碑。所有项目通用的偏好：语言、协作习惯、安全红线。
- 项目根目录的 `CLAUDE.md` —— 本项目的总则：目标、架构、硬约束、指向其他文档的索引。
- 子目录的 `CLAUDE.md` —— 各府衙的细则。Claude 只有在实际读那个目录下的文件时，才会把对应的 `CLAUDE.md` 懒加载进来。

![CLAUDE.md 三层布局](image/claude-md-layers.png)
*三层布局：家目录一块石碑，项目根一块石碑，子目录各有各的册子*

石碑短而有力，细则分而存放。和王国的律法，一模一样。

## 故事三：商人的一桩一单

城南有位姓张的商人，年轻时做的是杂货生意。所有账目都记在一本厚厚的大账册上，年头写到年尾，密密麻麻。

第五年上，一位旧主顾上门讨债，说三年前他曾让张商人代购一批丝绸，尾款尚未结清。张商人愣住——翻遍整本大账，竟找不出那一笔。不是没记，是大账册里横纵交错，再也认不出哪一行。最终只好自认倒霉，赔出一大笔银子。

事后老师傅过来看了账，摇头："一桩生意，一张单子。每桩单独开一张，按日编号，写明对方、货物、定金、交期。单子归档在抽屉里，索引记在一本小册子上。"

张商人依言做了。起初觉得麻烦——一桩五文钱的小生意也要开张单子？过了三年，他才知道老师傅的深意。有一次一位老主顾上门核对八年前一笔货款，他抽屉一翻，那张泛黄的单子前因后果清清楚楚，对方再无话说。

![商人的抽屉与编号单据](image/merchant-receipts.png)
*一桩生意一张单，按日编号，索引另记*

晚年他收弟子，只传一条："**账不怕多，怕不清。**"

### 揭示：decision.md 的升级路径

这个故事讲的是决策文档的演化——从一个 `decision.md` 升级到 **ADR**（Architecture Decision Record，架构决策记录）。

起步阶段，一个 `decision.md` 单文件就够。项目小，决策少，写几段话能翻得回来。

项目一旦上规模，你会遇到张商人年轻时的困境：翻不到旧决策、认不出关键一条、上下文全糊在一起。业界主流做法是**一决策一文件**，放在 `docs/adr/` 目录下顺序编号：

```
docs/adr/
├── 0001-choose-postgres-over-mysql.md
├── 0002-why-bun-not-node.md
├── 0003-auth-strategy.md
└── ...
```

每个文件独立，包含：决策内容、Why、备选方案、代价、日期。可以单独引用、可以跟 PR 走、可以随时新增一条而不动旧的。

AI 时代还演化出了 **AgDR**（Agent Decision Record），专门记录 AI agent 自己做的决策，格式一样，多了一个"这决定是 AI 做的还是人做的"的字段。

对个人项目，从单文件 `decision.md` 起步就好。规模上来、开始团队协作、决策需要和 PR 挂钩的那一天，再把它拆成 `docs/adr/`。不早，不晚。

**账不怕多，怕不清**。

## 故事四：三路镖师

某镖局接了一单大生意——把一批贵重瓷器从京城押运南下。这批货太显眼，总镖头反复权衡，决定分三队，走三条不同的路。

出发前一夜，总镖头把三位队长召到堂前。他没让三人凑在一起商议，反而一个接一个单独召见——每人一份路书，上面写着这一队的路线、暗号、接应人、遇险预案、到达后如何会合。

"你们三队路上不相见，也不互通消息。"总镖头说，"若出了事，各自按自己这份路书办。"

大队长问："若我遇险，另外两队怎么办？"

总镖头答："**我给你们的路书里，已经写好了。**你若遇险，某处有鸽信；鸽信若不至，另两队到某处便改道。——你不必指望临时传话。能指望的，只有出发前已经写明的。"

![三路镖师各走一路](image/three-caravans.png)
*三队各走一路，彼此不见面；全靠出发前那份路书*

三路均安然抵达。镖局从此立下规矩：分头办事，事前写明；能写在纸上的，不靠临时传话。

### 揭示：subagent 看不到主会话

这个故事讲的是 Claude Code 最容易被忽视的一个机制——**subagent 的上下文隔离**。

当 Claude 派出一个 subagent 去独立完成任务时，subagent 运行在它自己的上下文窗口里，和主会话彻底隔离。它**看不到**你在主会话里讲过的任何背景、听不到你跟主 Claude 的来回讨论。它只能看到你在派它出去时塞进它 prompt 的那些字，以及它自己主动读取的文件。

这意味着：**你希望 subagent 知道的一切，必须提前写在它能读到的地方**。

写在哪里？就是 `CLAUDE.md`、`plan.md`、`decision.md` 这一整套皮册里。Subagent 被派出之前，主 Claude 会把相关文件内容带进它的 prompt；或者 subagent 自己会在项目目录里扫一圈，把这些文件读进来。

没有这套皮册，subagent 就像没拿到路书的镖师——上路之后发生任何情况，它都只能凭当下那点信息瞎猜。

**能写在纸上的，不靠临时传话。**

## 故事五：守灯塔的老人

沿海有座孤岛，岛上一座灯塔。守塔人是一位独居的老人。

灯塔每夜必亮，海上每日天气必记。年轻时老人也偷过懒，想着"明天一起补"。结果有一次连补半个月，记下的都是大概——风是"大风"，浪是"高浪"，具体数据一概回忆不清。

那一季正逢海难，三艘船沉没。调查时来了官差要查他的日志，他拿不出任何精确的记录，脸上火辣辣。

自那以后，老人立下铁规矩：**灯熄即点，事发即记。从不拖到明日，从不写大概。**

每次入夜风起，他放下手中一切，到日志本上写下当时确切的时间、风向、风级、气压；每次船只经过，他记下船名、方向、吃水；哪怕只是一阵骤雨、一声远雷，他也即刻记下。

![守灯塔老人在灯下写日志](image/lighthouse-keeper.png)
*灯熄即点，事发即记；靠的不是记得清，是记得早*

几十年下来，守塔老人的日志本成了这一带航船最信赖的东西。渔民每出海前必来翻一翻，看去年同期这片海是什么样。

**靠的不是记得清，是记得早。**

### 揭示：文档维护的三条纪律

结构再好，纪律一烂，皮册就变成一堆僵尸文件。我见过太多项目，`progress.md` 停留在项目初始化那天，`decision.md` 里只有一条模板占位符，`handoff.md` 写的是三个月前的事。**这种文档还不如没有**，因为它会误导未来的 Claude 和未来的你。

我给自己和 Claude 立了三条硬规矩：

**第一，触发即更**。修完一个 bug，立刻把那条从 `bug.md` 的 OPEN 移到 FIXED，补现象、根因、修复、日期。做了一个架构决定，立刻在 `decision.md` 写一段。完成 `plan.md` 里的一个勾选项，立刻在 `progress.md` 追加一条。**不要攒**。人的记忆会美化、会遗忘——五分钟后还清晰的事，半小时后就变成"大概那样"。Claude 看到的模糊记忆，回头会变成它做错决定的根源。

**第二，日期用真实当天**。禁止"本周"、"最近"、"前几天"，一律写 `2026-04-24` 这样的 ISO 日期。文档是写给**未来**的你和未来的 Claude 看的——半年后你翻开 `progress.md`，看到"最近重构了鉴权模块"，"最近"是啥时候？无从考证。

**第三，改完直接 commit**。这三类文档的改动和代码一样重要，甚至更重要。每次更新都 commit，message 简短直白："update progress: finish auth refactor"。好处是 `git log` 本身就成了一条时间线，和 `progress.md` 互相印证，谁也骗不了谁。

灯塔的日志可贵，**不在老人记得清，在他记得早**。你的文档也是。

## 工具箱：一键生成皮册的 `/inpro` skill

五个故事讲完，剩下的是一个实操问题——每次新项目都手动建这 6 个文件？当然不。我把它做成了一个 Claude Code skill，名字叫 `inpro`（initialize project）。对话里打一句 `/inpro`，或者说"初始化项目"，Claude 会根据当前对话里的项目背景自动生成这 6 个文件，不留空模板，直接填上真实内容。

下面是完整的 skill 源码，复制到 `~/.claude/skills/inpro/SKILL.md` 直接可用：

````markdown
---
name: inpro
description: Initialize a project with standard Claude collaboration docs — creates CLAUDE.md, plan.md, progress.md, decision.md, bug.md, handoff.md in the current working directory (or the path given as argument). Use when the user says "inpro", "初始化项目", "做项目初始化", or explicitly invokes /inpro.
---

# inpro — 项目初始化

在目标目录下创建 6 个协作骨架文件：CLAUDE.md、plan.md、progress.md、decision.md、bug.md、handoff.md。

## 执行步骤

1. 确认目标目录（用户给路径就用，否则 cwd；不存在就 mkdir -p）
2. 检查冲突（同名 md 已存在则问用户：覆盖 / 跳过 / 合并）
3. 了解项目（对话里有主题就直接用；完全没有只问一句"这个项目是做什么的？"）
4. 按下方模板写入 6 个文件，**把项目名、目标、上下文填进去，不留空模板**
5. 如果目录不是 git 仓库，问一句要不要 git init + 首个 commit
6. 一句话汇报建了什么，在哪里

## CLAUDE.md 关键段落

# {项目名}
{一句话项目描述}

## 目标
{核心目标}

## 关键约束
- {从对话里提取的硬约束}

## 关键文件
- plan.md / progress.md / decision.md / bug.md / handoff.md

## 文档维护规则（主动执行，无需用户提醒）
- 修完一个 bug → bug.md：OPEN 那条移到 FIXED
- 完成勾选项或阶段性进展 → progress.md：追加一条
- 做了架构/选型决策 → decision.md：追加一条
- 讨论出新任务、scope 变化 → plan.md：改勾选项/加条目
- 会话即将结束 → 刷新 handoff.md
- 发现新 bug 未当场修 → bug.md OPEN 区追加

日期统一用真实当天日期。不要累积"等会儿一起更"，触发即更。
````

完整模板里还包含 `plan.md`、`progress.md`、`decision.md`、`bug.md`、`handoff.md` 五个文件各自的骨架。因为篇幅原因这里就不全贴了，核心思路都在上面的 `CLAUDE.md` 段落里——让 Claude 自己看到文档维护规则，并主动执行。

## 终章

五个故事讲完，回到开头那位国王。

他做的，不是治好那位智者的失忆，也不是换一个不会失忆的人——那样的人天下没有第二个。

他做的，只是在那张书案上，放了一本皮册。

我们这一代每一个写代码的人，都有机会召唤到自己的"天下第一聪明的谋士"。它真的很聪明，能在十分钟之内解掉你卡住三天的问题。

但它会失忆。每一次都失忆。

你可以每天重新讲一遍——讲到自己都烦。你也可以花十分钟，给它铺出一本皮册，再加上四条纪律：律法石碑要短、一桩生意一张单、镖师路书事前写明、灯塔日志事发即记。

从此之后，它依然每夜失忆，但你们的智慧，留在了皮册里。

> **聪明可以借用，记忆必须自备。**

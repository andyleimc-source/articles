# Vibe Coding 项目的六文件文档体系

**一套简洁的、为 AI 原生开发设计的文档结构，适配任意规模的项目。**

---

### 问题所在

大多数开发者要么过度文档化（写了几十页没人看的规格说明），要么完全不写文档（什么都没有，然后每次会话都要花 20 分钟重新给 AI 解释背景）。两种做法在 AI 辅助开发的场景下都行不通。

核心问题在于：**AI 需要在正确的时机获得正确数量的正确上下文。** 同样，人类也需要一份清晰的记录——项目到底是什么。

---

### 解决方案：六个文件，两类受众

每个项目固定使用以下六个文件，不多不少：

```
README.md       ← 给项目外部的人看
CLAUDE.md       ← 给 AI 看
PLAN.md         ← 项目走向
BUG.md          ← 踩过的坑
DECISIONS.md    ← 为什么这样设计
LESSONS.md      ← 不能忘的经验
```

最关键的分工：**README 给人看，CLAUDE.md 给 AI 看。** 两个文件从完全不同的角度描述同一个项目，内容绝不重复。

---

### 逐文件说明

#### `README.md` — 项目的门面

**受众：** 任何第一次接触这个项目的人——协作者、开源用户，或六个月后的你自己。

**用途：** 回答"这是什么、能做什么、怎么用？"不涉及内部实现细节，不写 AI 专用的上下文。就是一份干净、诚实的项目外部描述。

**写什么：**
- 一段话的项目描述
- 安装和配置步骤
- 使用示例
- 目录结构（可选，但有帮助）
- 链接到 PLAN.md 查看进度

**示例：**
```markdown
# HAP Auto Maker

AI 驱动的工具，将自然语言描述转化为完整的明道云 HAP 应用——
包括工作表、视图、工作流、模拟数据和权限——一条命令搞定。

## 快速开始
python3 scripts/run_app_pipeline.py
```

---

#### `CLAUDE.md` — AI 的简报

**受众：** AI 助手（Claude Code、Cursor 等）——每次会话开始时自动加载。

**用途：** 让 AI 立刻获得项目上下文，不需要你重复解释。与 README 本质不同：这是内部的、技术性的，是**写给 AI 的**，不是写给人的。

**保持简短。** 10–20 行是理想长度。每多一行都在消耗 context window。

**写什么：**
- 一句话项目概述（用于上下文，不是营销）
- 如何运行项目
- AI 必须遵守的关键规则、注意事项或约束
- API 响应格式或命名规范

**示例：**
```markdown
# CLAUDE.md — HAP Auto Maker

## 概述
将自然语言需求通过 AI 规划 + 多波次执行流水线转化为完整 HAP 应用。

## 运行
python3 scripts/run_app_pipeline.py

## 规则
- AI 调用一律使用 fast 档，禁用 reasoning
- 所有 AI 输出 → repair_plan() → validate_*() 后才能使用
- Web API 成功标志是 state == 1（不是 200，不是 0）
```

注意 CLAUDE.md 里没有的内容：安装指南、功能列表。那是 README 的职责。

---

#### `PLAN.md` — 路线图

**用途：** 项目目标、阶段划分和当前进度。"我们在做什么、做到哪里了"的唯一真实来源。

按需读取（不自动加载），所以可以写得详细一些。

```markdown
## 目标
构建统一的 MCP 服务器，暴露所有 HAP 平台操作。

## 第一阶段：核心 API（进行中）
- [x] 应用与导航组管理
- [x] 工作表视图
- [ ] 图表视图 ← 下一步

## 风险
- 私有 API 可能不经通知直接变更
```

---

#### `BUG.md` — Bug 记录

**用途：** Bug 及其修复方式的永久记录。防止几周后重新排查同一个问题。

每次解决一个非显而易见的 Bug 就写一条：

```markdown
## [BUG-001] creation_order 缺少工作表
- **状态**: 已解决
- **现象**: AI 工作表规划后验证时抛出 ValueError
- **根因**: AI 只列出了有依赖关系的工作表，未列出全部
- **修复**: 添加 repair_plan() 自动补全 + Prompt 约束
```

没有这个文件，你会在一个已经解决过的 Bug 上再花 45 分钟。

---

#### `DECISIONS.md` — 架构日志

**用途：** 记录关键决策**为什么**这样做。不是记录决定了什么——而是背后的推理。

很多项目后来走偏，是因为没人记得当初某个约束存在的原因。

```markdown
## [决策] 规划与执行分离

**决策：** 将 AI 规划脚本（plan_*）和执行脚本（create_*）拆开。

**原因：** 可以在不重新调用 AI 的情况下重跑执行（节省成本和时间）。
输出 JSON 可检查。故障可以定位到 AI 层还是 API 层。
```

---

#### `LESSONS.md` — 跨项目记忆

**用途：** 可以迁移到下一个项目的可复用知识。与 BUG.md 的区别在于：BUG.md 是项目内的，LESSONS.md 记录的是**下个项目也能用上**的东西。

```markdown
## 明道 Web API：成功标志是 state == 1，不是 200

## OAuth scope 错误返回业务码，不是 HTTP 401

## 浏览器自动化中导航后，需要重新注入拦截器
```

### 核心规则：README ≠ CLAUDE.md

| | README.md | CLAUDE.md |
|---|---|---|
| **受众** | 人 | AI |
| **语气** | 亲切，面向用户 | 技术性，直接 |
| **安装指南** | ✅ 有 | ❌ 无 |
| **功能描述** | ✅ 有 | ❌ 无 |
| **AI 规则与约束** | ❌ 无 | ✅ 有 |
| **API 响应格式** | ❌ 无 | ✅ 有 |
| **长度** | 按需 | 20 行以内 |

判断标准：如果第一次进来的人会觉得有用 → 放 README。如果只有在项目内部工作的 AI 才需要 → 放 CLAUDE.md。

---

### 规则清单

1. **严格六个文件。** 不要创建 `NOTES.md`、`DEBUG.md`、`RULES.md`。把这些内容合并到对应的标准文件里。
2. **不要混用。** README 给人，CLAUDE.md 给 AI。两者之间绝不重复内容。
3. **CLAUDE.md 保持精简。** 它每次会话都会加载。塞太多内容会损害 AI 表现。
4. **从最小集合开始，按需增长。** 新项目？先建 README + CLAUDE.md + PLAN.md，其余的等需要时再加。
5. **只管文档文件。** 六文件规则只适用于项目文档的 Markdown 文件。

---

### 从这里开始

新项目先建这三个文件：

**`README.md`** — 一段话：项目是什么、怎么运行。

**`CLAUDE.md`** — 三件事：项目概述（一行）、运行命令、关键规则。

**`PLAN.md`** — 目标和前三个任务。

其余的，边做边加。
